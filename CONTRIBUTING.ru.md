# Участие в разработке

Спасибо за улучшение Phenogram Bindings.
Делайте каждое изменение небольшим и проверяемым.

## Перед началом

Проверьте существующие issues.
Опишите большое изменение до начала реализации.
Используйте [официальную документацию Telegram Bot API](https://core.telegram.org/bots/api) как источник схемы.

Не добавляйте токен бота в тест, фикстуру, лог или коммит.

## Сгенерированные файлы

Большая часть файлов API, типов, сериализатора и фабрик генерируется.
Проект [`phenogram/scraper`](https://github.com/phenogram/scraper) содержит генератор.

Определите сгенерированные файлы перед редактированием.
Обновите генератор, если изменение схемы затрагивает много файлов.
Укажите версию Telegram Bot API, из которой взято изменение.

## Язык документации

Сохраняйте одинаковое содержание документов на английском и русском языках.
Обновляйте обе языковые версии в одном изменении.

Используйте контролируемый английский язык в английских документах:

- Используйте один термин для одного значения.
- Используйте короткие предложения.
- Используйте активный залог.
- Помещайте одну инструкцию в каждое предложение.
- Начинайте инструкцию с глагола в повелительном наклонении.
- Не используйте сокращение без объяснения.
- Не заменяйте термин синонимом только ради стиля.

Эти правила соответствуют назначению [ASD-STE100](https://www.asd-ste100.org/about_STE.html).
Имена API считаются допустимыми техническими терминами проекта.

## Примеры

Помещайте исполняемые скрипты в `examples`.
Каждый скрипт должен работать без токена бота.
Используйте фиксированные данные для каждого проверяемого значения.
Выбрасывайте исключение при неверном результате.
Добавляйте скрипт в `tests/Examples/ExamplesTest.php`.
Описывайте скрипт в обоих индексах примеров.

Не добавляйте рабочие примеры в стандартный набор тестов.

## Тесты

Установите зависимости для разработки:

```bash
composer install
composer tools:install
```

Запустите обязательные проверки:

```bash
for file in examples/[0-9][0-9]-*.php; do php "$file"; done
vendor/bin/phpunit
composer phpstan
composer lint:composer
```

Запустите проверку стиля:

```bash
tools/php-cs-fixer/vendor/bin/php-cs-fixer fix \
--dry-run \
--diff \
--sequential \
--config=.php-cs-fixer.php \
--allow-risky=yes
```

Объясняйте пропуск проверки в pull request.
Не скрывайте ошибку проверки.

## Pull requests

Используйте понятный заголовок.
Опишите видимый для пользователя результат.
Перечислите команды проверки.
Добавьте ссылку на соответствующее изменение Telegram.
Укажите изменение совместимости.
Не добавляйте несвязанное форматирование в diff.

Отправляя изменение, вы соглашаетесь с его распространением по лицензии MIT этого репозитория.
