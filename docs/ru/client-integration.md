# Интеграция клиента

Это руководство определяет транспортный контракт Phenogram Bindings.
Используйте его при реализации `ClientInterface`.

## Контракт интерфейса

Интерфейс содержит один метод:

```php
public function sendRequest(
    string $method,
    array $data,
): Phenogram\Bindings\Types\Interfaces\ResponseInterface;
```

`$method` содержит имя метода Telegram, например `sendMessage`.
`$data` содержит сериализованные имена полей Telegram.
Массив не содержит значений `null`.

Метод должен вернуть полную оболочку ответа.
Не возвращайте только значение `result`.

## Адрес запроса

Отправляйте запросы к облачному API по этому адресу:

```text
https://api.telegram.org/bot<token>/<method>
```

Используйте HTTPS для облачного API.
Кодируйте текст в UTF-8.
Не добавляйте токен бота в логи и сообщения исключений.

Telegram принимает запросы GET и POST.
Используйте POST в универсальном клиенте.
POST поддерживает JSON и загрузку файлов.

[Официальные правила запросов](https://core.telegram.org/bots/api#making-requests) являются главным источником.

## Кодирование запроса

Используйте `application/json`, если запрос не загружает файл.
Закодируйте весь массив `$data` как JSON.

Используйте `multipart/form-data`, если запрос загружает файл.
Преобразуйте каждый объект `InputFileInterface` в файловый объект вашей HTTP-библиотеки.
Кодируйте вложенные объекты и массивы Telegram как JSON-строки в multipart-полях.
Используйте значение `attach://<name>` для файла во вложенном медиаобъекте.
Добавьте файл как отдельную multipart-часть с тем же именем вложения.

Сериализатор сохраняет объекты `InputFileInterface` без изменений.
Ваш клиент определяет конкретные файловые классы.
Ваш клиент также определяет способ чтения каждого файла.

Не читайте непроверенный путь к файлу.
Установите ограничение размера перед чтением содержимого файла.

## Оболочка ответа

Telegram возвращает JSON-объект.
Объект всегда содержит `ok`.
Он также может содержать следующие поля:

| Поле | Значение |
|---|---|
| `result` | Результат успешного метода |
| `description` | Читаемый текст ошибки |
| `error_code` | Код ошибки Telegram |
| `parameters` | Структурированная информация для восстановления |

Декодируйте ответ в ассоциативный массив.
Проверьте, что декодированное значение является объектом.
Проверьте наличие `ok`.

Создайте `Phenogram\Bindings\Types\Response` из этих данных.
Создайте `ResponseParameters`, если поле `parameters` присутствует.
Сохраните декодированное значение `result` без изменений.

`Api` проверяет `ok` и `result`.
Он выбрасывает `ResponseException`, если `ok` равно `false` или `result` равно `null`.
Он передаёт успешное значение `result` в `Serializer`.

Запустите пример ошибки:

```bash
php examples/06-handle-api-error.php
```

## Категории ошибок

Отделяйте транспортные ошибки от ошибок Telegram.

Используйте транспортное исключение для следующих ошибок:

- Ошибка DNS
- Тайм-аут соединения
- Ошибка TLS
- Некорректный JSON
- Неожиданное тело HTTP-ответа

Возвращайте `ResponseInterface` для корректной оболочки ошибки Telegram.
Позвольте `Api` преобразовать эту оболочку в `ResponseException`.

Такое разделение позволяет приложению выбрать правильную политику повторов.

## HTTP-статус

Проверьте HTTP-статус перед декодированием тела.
Сохраните тело для корректной оболочки ошибки Telegram.
Не добавляйте токен бота в исключение.

Telegram может вернуть данные для восстановления в `ResponseParameters`.
Используйте `retryAfter` при ограничении частоты.
Используйте `migrateToChatId`, когда Telegram сообщает о миграции чата.

Привязки не выполняют повторы.
Политика повторов принадлежит приложению или клиенту.

## Тайм-ауты

Установите тайм-аут соединения.
Установите тайм-аут запроса.

Для `getUpdates` установите тайм-аут запроса больше аргумента Telegram `timeout`.
Добавьте небольшой запас для сети.

Не повторяйте неидемпотентный метод без политики приложения.
Повтор может создать дубликат сообщения.

## Локальный сервер Bot API

Сделайте базовый адрес API настраиваемым.
Эта настройка поддерживает [локальный сервер Bot API](https://core.telegram.org/bots/api#using-a-local-bot-api-server).

Не считайте ограничения файлов облачного и локального серверов одинаковыми.
Прочитайте официальную документацию сервера перед изменением ограничений загрузки.

## Доставка обновлений

Telegram поддерживает long polling и webhooks.
Эти режимы являются взаимоисключающими.

Используйте `getUpdates` для long polling.
Увеличивайте offset после каждого принятого обновления.

Используйте `setWebhook` для доставки через webhook.
Проверяйте заголовок `X-Telegram-Bot-Api-Secret-Token`, если настроили секретный токен.

Прочитайте [официальные правила обновлений](https://core.telegram.org/bots/api#getting-updates).

## Проверки для рабочей среды

- Скройте токен в логах.
- Проверяйте TLS-сертификаты.
- Установите тайм-ауты соединения и запроса.
- Ограничьте размер ответа.
- Ограничьте размер загрузки.
- Проверяйте локальные пути к файлам.
- Декодируйте JSON с включёнными исключениями.
- Сохраняйте `ResponseParameters`.
- Отделяйте транспортные исключения от `ResponseException`.
- Добавьте тесты успешного ответа, ошибки Telegram, некорректного JSON, тайм-аута и загрузки файла.

Используйте HTTP-клиент из [Phenogram Framework](https://github.com/phenogram/framework), если свой транспорт не нужен.
